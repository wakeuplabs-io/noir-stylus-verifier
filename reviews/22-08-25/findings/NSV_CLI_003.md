# NSV_CLI_003 — nsv generate panics on invalid file paths instead of graceful error handling

**Severity:** Low  
**Component:** CLI/Error Handling  
**Date:** 2025-01-16

## Description
The `nsv generate` command panics with an unwrap() when given invalid file paths, causing the application to crash instead of providing graceful error messages. This affects reliability and user experience.

## Impact
- **Application crash**: Panic causes immediate termination
- **Poor error handling**: Stack trace exposed instead of user-friendly message
- **Service interruption**: Invalid input can crash the CLI
- **Development friction**: Developers see scary stack traces for simple mistakes

## Likelihood
Medium - Users may provide incorrect paths during development.

## Preconditions / Assumptions
- User provides invalid file path to `--bytecode-path` or similar options
- Path points to non-regular file (e.g., directory, device file, etc.)

## Steps to Reproduce
1. Create project: `nsv new test`
2. Run with invalid bytecode: `nsv generate --bytecode-path "/dev/null"`
3. Expected: Graceful error message
4. Observed: Panic with stack trace

## Evidence
**Stack trace location:**
```
thread 'main' panicked at cli/src/infrastructure/system.rs:67:44:
called `Result::unwrap()` on an `Err` value: Error { 
    kind: InvalidInput, 
    message: "the source path is neither a regular file nor a symlink to a regular file" 
}
```

**Additional observations:**
- `--vk-path "/etc/passwd"` works (potential info disclosure)
- `--bytecode-path "../../../../etc/passwd"` fails gracefully

## Affected Scope
- `cli/src/infrastructure/system.rs:67` - Direct unwrap() call
- `nsv generate` command error handling
- Potentially other commands with file path inputs

## Recommendation
1. **Replace unwrap() with proper error handling:**
```rust
// Instead of:
some_operation().unwrap()

// Use:
some_operation().map_err(|e| AppError::FileError(format!("Invalid file path: {}", e)))?
```

2. **Validate file paths before processing:**
   - Check file exists and is readable
   - Validate file is regular file (not directory/device)
   - Consider path traversal protection

3. **Improve error messages:**
```
Error: Invalid bytecode file '/dev/null'
The file must be a regular file generated by nargo.
```

4. **Audit other unwrap() calls**: Search codebase for similar patterns

## Additional Considerations
- Path handling: Review if other inputs allow reading arbitrary files
- Information exposure: VK path reading system files may expose sensitive data
- Input validation: All file paths should be validated consistently

- Related to defensive programming practices
